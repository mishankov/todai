# AI-native task tracker

Статус: стартовая спецификация проекта  
Версия: 0.2  
Дата: 15 июля 2026 года

## 1. Кратко о проекте

Это personal-first таск-трекер, вдохновлённый Todoist, но изначально спроектированный с учётом работы встроенного ИИ-агента и внешних пользовательских агентов.

Первый релиз остаётся именно таск-трекером. Заметки, документация, полноценное рабочее пространство и прочие сущности могут появиться позже, но не входят в первоначальный scope.

Проект решает сразу две личные задачи автора:

1. Создать таск-трекер, которым хочется пользоваться и который можно развивать под собственный workflow.
2. Развивать и проверять на реальном продукте Go-фреймворк [Platforma](https://github.com/platforma-dev/platforma), уделяя особое внимание качеству реализации, архитектуре и developer experience.

Рыночный взрыв не является критерием успеха первой версии. Главная ценность проекта на старте — полезный личный продукт и качественно сделанная инженерная система.

## 2. Продуктовая идея

Обычный таск-трекер хранит задачи и исполняет явные команды пользователя. Этот продукт дополнительно предоставляет агенту безопасный и хорошо спроектированный интерфейс к задачам.

ИИ здесь не отдельная декоративная кнопка и не генератор текста. Он должен понимать текущее состояние задач, находить нужные данные, предлагать согласованные изменения и помогать управлять системой через диалог.

Предполагаются два способа агентного взаимодействия:

- встроенный серверный агент внутри продукта;
- внешние агенты пользователя, вызывающие публичный API или CLI таск-трекера.

Сначала реализуется встроенный агент. Публичная интеграция для внешних агентов появится после стабилизации доменной модели и API.

## 3. Цели первой версии

- Дать пользователю быстрый и приятный способ создавать, организовывать и завершать задачи.
- Защитить приложение полноценным входом по логину и паролю, даже в однопользовательском MVP.
- Реализовать встроенного серверного агента для работы с задачами на естественном языке.
- Разрешать агенту читать и изменять состояние таск-трекера через ограниченный набор доменных операций.
- Сохранять единый журнал всех продуктовых событий с указанием инициатора и источника.
- Использовать проект как основной dogfood-сценарий для Platforma.
- Сохранить достаточно простую архитектуру, которую можно целиком понимать и контролировать.

## 4. Что пока не делаем

- Не строим coding agent и среду удалённого исполнения кода.
- Не запускаем рабочие пространства, Docker-контейнеры или виртуальные машины для пользовательских репозиториев.
- Не создаём платформу эвалов для агентов.
- Не делаем MCP основой интеграционной архитектуры.
- Не реализуем заметки, документы, wiki, календарь и универсальную базу знаний в MVP.
- Не строим оркестратор множества автономных агентов.
- Не даём встроенному агенту shell, файловую систему или доступ к произвольной сети.
- Не разрабатываем собственный универсальный LLM runtime, пока Pi решает эту задачу.
- Не реализуем повторяющиеся задачи в MVP.
- Не реализуем отдельную систему подтверждения агентных операций в MVP.

## 5. Продуктовые принципы

### Personal-first

Первая версия создаётся прежде всего для автора. Она может быть однопользовательской, но доменная модель не должна намеренно блокировать последующий multi-user режим.

### Tasks first

Сначала нужно сделать хороший таск-трекер без необходимости оправдывать каждую функцию наличием ИИ. Агент усиливает работоспособную модель задач, а не заменяет её.

### AI-native inside and outside

Внутренний агент является первым приоритетом. Позже те же application services должны стать доступны внешним агентам через стабильный API и/или CLI.

### Explicit agent capabilities

Агент получает только явно разрешённые task-specific tools. Все операции проходят через те же Go application services и доменные проверки, что и действия из UI. Они записываются в общий журнал продуктовых событий наравне с действиями пользователя и системы. UX подтверждения отдельных операций будет спроектирован позже и не входит в MVP.

### Craftsmanship

Craftsmanship относится прежде всего к процессу и результату разработки самого продукта: ясным границам компонентов, качественным API, тестируемости, наблюдаемости и аккуратному UX.

### Boring infrastructure where possible

PostgreSQL и обычные процессы предпочтительнее дополнительной распределённой инфраструктуры, пока масштаб проекта не требует обратного.

## 6. Рабочий scope MVP

Ниже приведён минимальный продуктовый scope, достаточный для начала разработки. Детали модели задач ещё могут уточняться.

### Задачи

Для задачи предполагаются следующие поля:

- заголовок;
- необязательное описание;
- статус: активна или завершена;
- проект;
- необязательный раздел проекта;
- необязательная родительская задача;
- приоритет;
- необязательная дата исполнения;
- необязательное время исполнения, задаваемое только вместе с датой;
- позиция для ручной сортировки;
- время создания и изменения;
- автор последнего изменения.

### Организация

- Inbox для задач без выбранного проекта.
- Проекты.
- Разделы внутри проектов.
- Подзадачи.
- Представление всех задач.
- Представления Today и Upcoming.
- Два представления проекта: вертикальный список разделов и горизонтальная доска с колонкой для каждого раздела.
- Общий для обоих представлений ручной порядок разделов и задач.
- Drag-and-drop разделов и задач внутри раздела и между разделами в обоих представлениях проекта.
- Быстрый поиск.

Метки, сохранённые фильтры и совместная работа полезны, но не считаются обязательными для первого вертикального среза. Повторяющиеся задачи сознательно отложены: в текущем личном workflow они используются редко.

### Базовые пользовательские действия

- быстро создать задачу;
- изменить поля задачи;
- завершить и восстановить задачу;
- перенести задачу в другой проект;
- создать, переименовать, удалить и упорядочить раздел проекта;
- переместить задачу внутри раздела или в другой раздел с помощью drag-and-drop;
- переключить проект между представлениями списка и доски без изменения состава и порядка задач;
- создать и упорядочить подзадачи;
- открыть Inbox, все задачи, проект, Today или Upcoming;
- найти задачу;
- увидеть историю существенных изменений.

### Встроенный агент

Первые полезные сценарии:

- разобрать Inbox;
- найти задачи по смысловому запросу;
- показать, что требует внимания сегодня;
- предложить план на день;
- декомпозировать задачу на подзадачи;
- изменить приоритеты и сроки нескольких задач;
- перенести задачи между проектами;
- объяснить, какие изменения он предлагает и почему.

### Поверхности взаимодействия с агентом

У агента будет отдельный полноценный чат. Кроме него, в интерфейсе появятся контекстные точки входа в стратегически выбранных местах:

- кнопки действий рядом с задачей, проектом или представлением;
- компактные поля ввода для команды в контексте текущего экрана;
- готовые действия вроде «разобрать Inbox», «декомпозировать задачу» или «спланировать день».

Контекстная точка входа должна передавать агенту явный структурированный контекст: идентификатор задачи, проекта или представления. Точное размещение, формат чата и переходы между контекстным действием и отдельным чатом требуют UX-прототипирования до реализации агентного UI.

## 7. Зафиксированный технологический стек

### Backend

- Go.
- [Platforma](https://github.com/platforma-dev/platforma).
- HTTP API.
- Server-Sent Events для потоковых событий агента.
- Полноценная авторизация по логину и паролю через домены `auth` и `session` Platforma.

MVP остаётся однопользовательским: публичной регистрации и управления несколькими аккаунтами нет, но все пользовательские endpoints защищены серверной сессией. Единственный аккаунт создаётся при первоначальной настройке или административной bootstrap-командой. Login создаёт серверную сессию и устанавливает защищённую cookie; logout удаляет сессию.

### Данные

- PostgreSQL.
- Миграции через Platforma.
- Доступ к PostgreSQL через существующий стек Platforma: `sqlx` и `lib/pq`.

### Клиенты

- SvelteKit для веб-приложения.
- Electron для десктопного приложения.
- Веб- и десктопный клиенты должны переиспользовать один UI и один HTTP API.

Практический порядок реализации: сначала веб-клиент, затем Electron shell после стабилизации основных экранов.

### Внутренний агент

- [Pi](https://pi.dev/) как agent runtime и слой интеграции с LLM-провайдерами.
- Небольшой собственный TypeScript runner поверх Pi SDK.
- Взаимодействие Go и runner через JSONL RPC по `stdin/stdout`.

Важно: протокол Pi называется RPC, но не является стандартным JSON-RPC 2.0. Каждая команда, ответ или event передаётся отдельным JSON-объектом в отдельной строке.

        │   │                      │
        │   │ • Pi AgentSession    │
        │   │ • provider auth      │

        │              │ HTTPS

### Ответственность Go-бэкенда

Go является архитектурным центром продукта и владельцем состояния.

Backend отвечает за:

- пользователей и авторизацию в продукте;
- задачи, проекты и представления;
- валидацию и права доступа;
- agent sessions и agent runs;
- единый журнал продуктовых событий;
- запуск, мониторинг и остановку процессов Pi;
- преобразование событий Pi в API продукта;
- хранение канонических данных в PostgreSQL.

Pi не обращается к PostgreSQL напрямую.

### Ответственность pi-runner

Runner отвечает за:

- создание и настройку `AgentSession`;
- выбор модели и провайдера;
- OAuth credentials провайдеров и обновление токенов;
- system prompt агента;
- регистрацию разрешённых task-specific tools;
- выполнение agent loop;
- streaming событий через JSONL RPC;
- cancellation, retry и context compaction.

Runner не содержит бизнес-правил таск-трекера.

### Почему нужен собственный runner

Обычный `pi --mode rpc` ориентирован на coding-agent сценарий. Наш runner должен:

- отключить встроенные `read`, `write`, `edit`, `bash` и другие coding tools;
- зарегистрировать только инструменты таск-трекера;
- задать собственный system prompt;
- зафиксировать версию Pi и изолировать Go-код от изменений его внутреннего API;
- при необходимости расширить RPC командами авторизации и диагностики.

## 9. Интеграция Go и Pi

В Go должен существовать внутренний интерфейс, не зависящий от конкретных типов Pi:

```go
type AgentRuntime interface {
	Start(ctx context.Context, session Session) error
	Prompt(ctx context.Context, message Message) (<-chan Event, error)
	Abort(ctx context.Context) error
	Close() error
}
```

Первая реализация — `PiRuntime`.

Пример команды Pi:

```json
{"id":"request-42","type":"prompt","message":"Разбери мой Inbox"}
```

Пример потокового события:

```json
{
  "type": "message_update",
  "assistantMessageEvent": {
    "type": "text_delta",
    "delta": "В Inbox находится "
  }
}
```

Go не должен передавать сырые события Pi напрямую клиентам. Адаптер преобразует их в стабильные события продукта, например:

- `agent.run.started`;
- `agent.message.delta`;
- `agent.tool.started`;
- `agent.tool.completed`;
- `agent.task.changed`;
- `agent.run.completed`;
- `agent.run.failed`.

Это позволит заменить или обновить runtime без изменения публичного API.

## 10. Модель процессов Pi

Начальная модель: один процесс Pi на активную агентную сессию.

```text
agent session A ── pi process A
agent session B ── pi process B
```

Для personal-first продукта обычно будет активен один процесс.

Процесс:

1. Запускается по первому сообщению.
2. Сохраняет контекст сессии между сообщениями.
3. Может быть остановлен после периода бездействия.
4. Восстанавливается при следующем сообщении.
5. Принудительно завершается при зависании или нарушении лимитов.

Go supervisor должен контролировать:

- startup timeout;
- максимальную продолжительность одного run;
- максимальное число model turns и tool calls;
- idle timeout;
- корректное завершение;
- аварийный restart;
- отсутствие мусора в `stdout`, нарушающего JSONL framing.

Логи runner должны идти в `stderr`.

## 11. Инструменты встроенного агента

Минимальный набор read tools:

- `task_search` — поиск задач с фильтрами;
- `task_get` — получение задачи и её контекста;
- `project_list` — список проектов;
- `view_query` — получение Inbox, Today, Upcoming или проектного списка.

Минимальный набор mutation tools:

- `task_create` — создать задачу или подзадачу;
- `task_update` — изменить поля задачи;
- `task_complete` — завершить задачу;
- `task_reopen` — восстановить задачу;
- `task_move` — переместить задачу в проект или под другую задачу;
- `task_reorder` — изменить порядок задач.

Явные инструменты упрощают JSON Schema, авторизацию, диагностику и последующее добавление разных политик подтверждения для разных операций. Для массовых сценариев позднее можно добавить типизированный batch tool, но он не должен обходить обычные доменные проверки.

Инструменты runner вызывают закрытые endpoints Go-бэкенда. Для каждого run выдаётся короткоживущий внутренний токен, ограниченный конкретным пользователем, сессией и набором разрешений.

## 12. Безопасность агентных изменений

В MVP отдельной системы preview/approve/reject нет. Вызов разрешённого mutation tool выполняет операцию сразу:

```text
Pi calls task_update
          │
          ▼
Runner calls internal Go API
          │
          ▼
Go authorizes and validates command
          │
          ▼
Application service updates PostgreSQL
          │
          ▼
Event records actor, source and run
```

Требования:

- runner не обращается к таблицам напрямую;
- каждая операция проходит через те же application services, что и обычный UI-запрос;
- внутренний токен ограничен пользователем, agent session, run и набором tools;
- в общем event log различаются пользователь, встроенный агент, внешний агент и система;
- tool result содержит фактически выполненное изменение, а не только сообщение об успехе;
- обновления используют `version` или другой механизм optimistic concurrency;
- при конфликте агент получает явную ошибку и должен перечитать задачу перед повтором;
- массовая операция использует транзакцию, если частичное выполнение нарушает её смысл.

Систему подтверждений нужно спроектировать отдельно на основе реального UX. Архитектура tools не должна ей препятствовать: позднее policy layer сможет пропустить операцию сразу, запросить подтверждение или запретить её в зависимости от типа действия и контекста.

## 13. Предварительная модель данных

### Пользовательские данные

```text
users
projects
project_sections
tasks
activity_events
```

Предполагаемые ключевые поля `projects`:

```text
id
user_id
name
layout
created_at
updated_at
```

`layout` принимает значения `list` или `board` и задаёт выбранное представление проекта. Представление не создаёт отдельную структуру или отдельный порядок задач.

Предполагаемые ключевые поля `project_sections`:

```text
id
user_id
project_id
name
position
version
created_at
updated_at
```

Предполагаемые ключевые поля `tasks`:

```text
id
user_id
project_id nullable
section_id nullable
parent_id nullable
title
description nullable
status
priority
due_date nullable
due_time nullable
due_timezone nullable
position
version
completed_at nullable
created_at
updated_at
```

Поле `version` или эквивалент необходимо для optimistic concurrency между UI и агентом.

Раздел принадлежит ровно одному проекту. `section_id` задачи может ссылаться только на раздел её проекта. Задачи без раздела остаются в проекте и отображаются в общей группе без раздела. При удалении раздела его задачи не удаляются, а при переносе задачи в другой проект её прежний раздел очищается. `position` определяет единый ручной порядок, используемый в обоих представлениях проекта.

### Общий журнал событий

`activity_events` — append-only журнал всех доменных, lifecycle- и security-событий независимо от их инициатора. Он включает не только действия агента, но и действия пользователя и системы. События не отбрасываются из-за того, что сейчас для них нет пользовательского UI.

Примеры типов событий:

- `auth.login.succeeded`, `auth.login.failed`, `auth.logout`;
- `task.created`, `task.updated`, `task.completed`, `task.reopened`, `task.moved`, `task.reordered`;
- `project.created`, `project.updated`, `project.archived`;
- `agent.session.created`, `agent.message.created`;
- `agent.run.started`, `agent.run.completed`, `agent.run.failed`, `agent.run.aborted`;
- `provider.connected`, `provider.disconnected`.

Предварительные поля события:

```text
id
type
occurred_at
actor_type
actor_id nullable
source
aggregate_type nullable
aggregate_id nullable
correlation_id
agent_run_id nullable
payload jsonb
```

Для изменения доменных данных событие записывается в той же PostgreSQL-транзакции, что и само изменение. Payload должен содержать достаточно данных для activity UI и диагностики, но не должен копировать пароли, OAuth credentials, reasoning или другие секреты.

Этот журнал не превращает приложение в event-sourced систему: актуальное состояние по-прежнему хранится в обычных доменных таблицах. `activity_events` служит для пользовательской истории, аудита, диагностики и будущих интеграций.

### Агентные данные

```text
agent_sessions
agent_messages
agent_runs
agent_run_events
provider_connections
```

Назначение:

- `agent_sessions` — пользовательские диалоги с агентом;
- `agent_messages` — каноническая история для UI;
- `agent_runs` — отдельные запуски модели;
- `agent_run_events` — значимые события и диагностика run;
- `provider_connections` — метаданные подключённых LLM-провайдеров без выдачи секретов клиенту.

Pi может использовать собственные session-файлы для восстановления внутреннего контекста, но они не являются источником истины для пользовательского интерфейса.

## 14. Предварительный HTTP API

Публичные имена endpoints ещё не зафиксированы, но API должен покрывать следующие ресурсы.

### Product authentication

```text
POST   /api/auth/login
POST   /api/auth/logout
GET    /api/auth/me
POST   /api/auth/change-password
```

Endpoint публичной регистрации в MVP не публикуется. Все остальные `/api` endpoints, кроме login и технического health check, требуют действующую session cookie.

### Tasks and projects

```text
GET    /api/tasks
POST   /api/tasks
GET    /api/tasks/{id}
PATCH  /api/tasks/{id}
POST   /api/tasks/{id}/complete
POST   /api/tasks/{id}/reopen
POST   /api/tasks/{id}/reorder

GET    /api/projects
POST   /api/projects
PATCH  /api/projects/{id}
GET    /api/projects/{id}/sections
POST   /api/projects/{id}/sections
PATCH  /api/projects/{id}/sections/{sectionId}
DELETE /api/projects/{id}/sections/{sectionId}
POST   /api/projects/{id}/sections/{sectionId}/reorder

GET    /api/views/inbox
GET    /api/views/today
GET    /api/views/upcoming
```

### Agent

```text
POST   /api/agent/sessions
GET    /api/agent/sessions/{id}
POST   /api/agent/sessions/{id}/messages
GET    /api/agent/sessions/{id}/events
POST   /api/agent/runs/{id}/abort
```

Отправка сообщения создаёт `agent_run`. Поток результата доставляется клиенту через SSE. Переподключение к SSE не должно терять уже сохранённые события.

### External agents — позже

После стабилизации внутреннего API:

- personal access tokens или OAuth для внешних клиентов;
- документированный REST API;
- Go CLI;
- идемпотентные mutation endpoints;
- actor attribution для внешнего агента.

## 15. Авторизация LLM-провайдеров

Основная причина выбора Pi — поддержка подписочной авторизации, прежде всего ChatGPT Plus/Pro через Codex, а также других поддерживаемых провайдеров.

### Personal MVP

Допустим ручной bootstrap:

1. Выполнить login Pi на сервере.
2. Сохранить `auth.json` в persistent storage.
3. Запускать runner с тем же изолированным `agentDir`.
4. Не передавать OAuth tokens в браузер или Electron.

### Multi-user версия

Потребуется web-facing flow:

- начать OAuth/device-code авторизацию;
- показать пользователю URL и код;
- принять callback или дождаться device confirmation;
- отобразить подключённый аккаунт;
- отключить аккаунт;
- изолировать credentials каждого пользователя.

У Pi нет готового HTTP auth server уровня OpenCode, поэтому эту часть должен будет предоставить наш runner или отдельный auth adapter.

Refresh tokens являются критическими секретами. Они должны быть зашифрованы на диске или храниться в предназначенном для секретов хранилище. Простое сохранение общего `auth.json` допустимо только для локального personal MVP.

## 16. Очереди и фоновые задачи

На старте отдельный брокер сообщений не требуется.

Предлагаемая последовательность:

1. Один экземпляр Go backend управляет локальными Pi-процессами.
2. Состояние `agent_runs` хранится в PostgreSQL.
3. Platforma queue можно использовать для локального выполнения фоновых задач, если это улучшит её dogfooding.
4. PostgreSQL-backed очередь вводится только перед несколькими backend replicas или при необходимости надёжного восстановления runs.

Redis, RabbitMQ и Kafka в первоначальную архитектуру не входят.

## 17. Наблюдаемость и надёжность

Для каждого agent run нужно сохранять:

- runtime и версию Pi;
- provider и model;
- время начала и завершения;
- итоговый статус;
- количество model turns и tool calls;
- token usage, если провайдер его возвращает;
- ошибки provider, runner и tools;
- выполненные task mutations и их итог;
- actor, source и correlation ID.

Не следует сохранять полные prompts, reasoning и tool payloads в обычные application logs. Пользовательские данные и credentials должны редактироваться или исключаться.

Необходимые защитные ограничения:

- максимальная продолжительность run;
- максимальное число шагов;
- максимальный размер контекста;
- cancellation через UI;
- retry только для временных ошибок;
- circuit breaker или cooldown при постоянных provider errors;
- graceful shutdown Go backend и runner.

## 18. Тестирование

### Backend

- unit tests доменных правил;
- integration tests PostgreSQL repositories и migrations;
- HTTP contract tests;
- tests login, logout, session expiration и защиты endpoints;
- tests optimistic concurrency и атомарности массовых операций;
- tests принадлежности разделов проекту, удаления раздела и переноса задач между разделами;
- tests прав доступа, actor attribution и записи всех доменных событий;
- tests атомарности изменения данных и соответствующего `activity_event`.

### Pi integration

- fake runner, который воспроизводит детерминированный JSONL stream;
- protocol conformance tests для parser и process supervisor;
- tests partial lines, invalid JSON, unexpected process exit и cancellation;
- end-to-end smoke test с реальным Pi, не обязательный для каждого локального test run.

### Frontend

- component tests для основных task flows;
- component tests представлений проекта и общего drag-and-drop порядка;
- tests отдельного чата и контекстных точек входа;
- end-to-end happy path: создать разделы, упорядочить задачи в List и Board и попросить агента изменить задачу.

### Platforma dogfooding

Во время разработки отдельно фиксируются:

- отсутствующие возможности фреймворка;
- неудобные API;
- повторяющийся boilerplate;
- сложные для тестирования границы;
- проблемы документации и диагностики.

Проблемы Platforma следует исправлять в Platforma, а не бессрочно обходить локальными product-specific абстракциями.

### Этап 0. Скелет

## 20. Definition of first usable version

Первая версия считается пригодной для личного использования, когда пользователь может:

1. Войти в приложение по логину и паролю и завершить сессию через logout.
2. Создать проект и несколько задач через веб-интерфейс.
3. Создать и упорядочить разделы проекта.
4. Переключаться между List и Board, сохраняя одинаковый состав и порядок разделов и задач.
5. Перетаскивать разделы и задачи внутри раздела и между разделами в обоих представлениях проекта.
6. Работать с Inbox, всеми задачами и Today.
7. Завершать и переносить задачи.
8. Открыть отдельный чат со встроенным агентом.
9. Попросить агента разобрать Inbox или декомпозировать задачу.
10. Увидеть потоковый ответ.
11. Увидеть корректно выполненные агентом изменения и соответствующие записи в общем журнале событий.
12. Запустить подходящее агентное действие из контекста задачи или представления.
13. Перезапустить сервис без потери задач и истории диалога.

## 21. Зафиксированные архитектурные решения

- Продукт начинается с таск-трекера, а не с универсального workspace.
- Backend пишется на Go и Platforma.
- Основная база данных — PostgreSQL.
- UI — SvelteKit, desktop — Electron.
- Встроенный агент работает на сервере.
- Agent runtime — Pi.
- Go взаимодействует с Pi через JSONL RPC по `stdin/stdout`.
- Используется собственный небольшой TypeScript runner поверх Pi SDK.
- Pi получает только task-specific tools и не получает shell/filesystem tools.
- Go backend остаётся владельцем бизнес-логики и данных.
- Внутренний агент реализуется раньше интеграций с внешними агентами.
- Внешние агенты в будущем работают через API и/или CLI продукта.
- PostgreSQL является источником истины; runtime-файлы Pi — только его внутреннее состояние.
- MVP является single-user, но использует полноценную авторизацию по логину и паролю и серверные сессии.
- Публичной регистрации пользователей в MVP нет.
- Повторяющихся задач в MVP нет.
- Разделы являются общей структурой проекта для представлений List и Board.
- List и Board используют единый ручной порядок разделов и задач; переключение представления не изменяет данные.
- Drag-and-drop доступен в обоих представлениях проекта.
- Отдельной системы подтверждения агентных операций в MVP нет.
- Агент изменяет задачи только через явно разрешённые tools и Go application services.
- Основные поверхности агента — отдельный чат и контекстные точки входа в task UI.
- Все продуктовые события записываются в единый append-only журнал независимо от инициатора.

## 22. Рабочие допущения, требующие подтверждения

Эти решения приняты для того, чтобы можно было начать разработку, но пока не были явно подтверждены:

1. Inbox, все задачи, проекты с разделами, подзадачи, Today и Upcoming составляют достаточное task-ядро MVP.
2. Метки и сохранённые фильтры можно отложить.
3. Первым подписочным провайдером будет OpenAI через ChatGPT/Codex login.
4. Веб-клиент реализуется раньше Electron-приложения.
5. Первый deployment может состоять из одного Go backend, PostgreSQL и локально управляемых им процессов runner без внешнего брокера очередей.

## 23. Открытые вопросы

Эти вопросы не блокируют создание скелета, но должны быть решены до соответствующего этапа:

- Какое имя получит продукт?
- Должен ли due date быть отдельным от schedule date?
- Какую часть общего event log показывать пользователю в activity UI, сохраняя полный журнал для аудита?
- Как устроить историю отдельного чата: один непрерывный диалог, несколько сессий или автоматическое разделение по периодам?
- Где именно разместить контекстные кнопки и поля ввода и как показывать передаваемый агенту контекст?
- Для каких операций в будущем потребуется подтверждение и как оно должно выглядеть, не прерывая естественный workflow?
- Храним ли provider credentials в зашифрованном PostgreSQL, файловом хранилище или отдельном secret store?
- Будет ли продукт self-hosted, hosted или сначала только локальной личной установкой?
- Будут ли backend, frontend и runner жить в одном monorepo?
- Как создаётся единственный пользователь: first-run setup screen, CLI bootstrap или переменные окружения?
